# 单包渐进迁移

## 适用场景

适用于当前仍是 `src/electron + src/renderer` 的单包项目。目标不是一次性重构成 workspace，而是先把边界收紧，再决定是否升级成单仓多包。

## 分阶段步骤

### 第一阶段：先收紧边界

- 把主进程能力归位到 `src/electron/core`、`ipc`、`services`。
- 把渲染层公共 UI、业务模块、页面入口规范化。
- 新增共享契约目录，例如 `src/shared` 或 `src/electron/shared`，用于存放 IPC 常量和纯函数。

### 第二阶段：收敛桥接面

- 统一由 preload 暴露白名单 API。
- 页面不再直接拼接 channel，也不再直接引用主进程路径。
- 把 `window` 能力访问封装成渲染层适配器。

### 第三阶段：业务域拆分

- 把大页面拆到模块目录，至少完成 `api / store / components / pages` 四层分工。
- 只在局部业务域内保留局部状态，避免“全局 store 巨石化”。

### 第四阶段：再决定是否升级为 workspace

- 当主进程依赖、预加载桥接、渲染层构建链已经明显膨胀时，再迁移到 `pnpm workspaces`。
- 迁移时优先抽 `shared`，其次抽 `preload`，最后拆 `main` 与 `renderer`。

## 停靠点

- 第一阶段完成后，项目仍然可以保持单包结构，不必继续搬迁。
- 只要 `renderer` 已经不再越权引用主进程代码，就已经获得最大收益的一半以上。

## 回滚边界

- 每个阶段都应保持可运行。
- 任何阶段如果需要大规模改 import 路径，先引入薄适配层，再逐步替换。
- 不要同时做目录搬迁、状态重写、UI 重构三件事。
